# SOP Execution Verification

## Overview

SOP 执行完成后，生成执行记录（步骤、参数、产出），然后由专门 agent 群体做审查，结果返回报告。

核心思路是**执行 → 记录 → 多视角审查 → 报告**：

```
┌──────────────────────────────────────┐
│          SOP 执行完成                  │
│  收集: state / 产出文件 / 执行日志     │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│        SOP Verification Engine       │
│  加载预期产出定义 → 收集实际产出      │
│  对比 → 生成验证基线报告              │
└──────┬──────────────────────┬───────┘
       │                      │
       ▼                      ▼
┌──────────────┐    ┌──────────────────┐
│ 基础验证     │    │ 多 Agent 深度审查 │
│ · 步骤合规   │    │ · 流程合规 Agent │
│ · 产出完整   │    │ · 质量审查 Agent │
│ · 状态一致   │    │ · 安全审查 Agent │
│ · 参数一致   │    │ · 代码审查 Agent │
└──────┬───────┘    │ · 架构审查 Agent │
       │            └────────┬─────────┘
       │                     │
       ▼                     ▼
┌──────────────────────────────────────┐
│         SOP 审查总报告               │
│  ✅/⚠️/❌ 各维度评分                 │
│  反模式清单 + 建议                   │
└──────────────────────────────────────┘
```

## 工作流程

### Step 1: 收集执行上下文 [AUTO]

自动收集以下信息：
- `.sop/state/{sop}-{date}.json` — 状态文件（步骤、参数、状态）
- `.sop/output/` — 产出文件列表
- **预期产出定义** — 从 SOP 配置或 `sop-expected.yml` 读取

### Step 2: 加载预期产出定义 [AUTO]

读取预期产出定义，了解：
- 应该生成哪些文件
- 应该执行哪些步骤
- 步骤间参数如何传递

### Step 3: 基础验证 [AUTO]

引擎自动执行 4 项基础检查：

| 检查项 | 检测内容 |
|--------|----------|
| **步骤合规** | 执行步骤与 step-map 定义顺序一致？有无跳过/重复？ |
| **产出完整性** | 预期产出文件是否存在？内容非空？ |
| **状态一致性** | state 标记 completed 的步骤是否确实有对应产出？ |
| **参数一致性** | 用户输入的参数是否被正确使用？有无丢失？ |

### Step 4: 多 Agent 深度审查 [AUTO]

引擎将上下文分发给各审查 Agent：

| Agent | 视角 | 产出 |
|-------|------|------|
| 流程合规审查 | 步骤顺序、流转是否符合规范 | 流程合规报告 |
| 质量审查 | 产出内容质量、完整性 | 质量评估报告 |
| 安全审查 | 敏感信息泄露、安全风险 | 安全审计报告 |
| 代码审查 | 代码规范、潜在 bug | 代码审查报告 |
| 架构审查 | 架构合理性、技术选型 | 架构评估报告 |

### Step 5: 生成总报告 [AUTO]

聚合所有审查结果，生成总报告：

```markdown
# SOP 审查报告: ${sop-name}

## 执行概要
- **SOP**: xxx
- **执行时间**: 2026-05-21
- **总体评分**: 85/100 ⚠️

## 基础验证
| 维度 | 状态 | 详情 |
|------|------|------|
| 步骤合规 | ✅ | 8/8 步骤按序执行 |
| 产出完整性 | ⚠️ | 缺少: api-docs.md |
| 状态一致性 | ✅ | 状态与实际产出匹配 |
| 参数一致性 | ✅ | 参数使用正确 |

## 深度审查
### 流程合规 ⚠️
- 步骤 3→4 之间预期有确认环节，实际未确认

### 质量审查 ✅
- 文档结构清晰，覆盖主要功能点

### 安全审查 ✅
- 未发现硬编码密钥或敏感信息泄露

### 代码审查 ⚠️
- 发现了 3 个 linting 警告

### 架构审查 ✅
- 整体架构合理，无重大架构问题

## 反模式汇总
| # | 类型 | 严重度 | 描述 |
|---|------|--------|------|
| 1 | 步骤跳过 | MEDIUM | 步骤 2 被跳过 |
| 2 | 产出缺失 | LOW | api-docs.md 未生成 |

## 建议
- 补充 api-docs.md 产出
- 在步骤确保确认环节不被跳过
```

## 预期产出定义

每个 SOP 可在 `.claude/skills/{sop-name}/expected.yml` 中定义预期产出：

```yaml
sop: code-review
expected_steps:
  - id: 1_scope
    name: 理解变更
  - id: 2_review
    name: 代码审查
  - id: 3_test
    name: 运行测试
  - id: 4_report
    name: 审查报告

expected_outputs:
  - path: .sop/output/code-review-*.md
    description: 审查报告
    type: markdown
    required: true
  - path: tests/**/test-*.java
    description: 新增测试
    type: code
    required: false
```

详见 `references/DSL.md`。
